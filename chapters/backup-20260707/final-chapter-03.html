🔗 星恒项目 · Agent 协作全景

以下展示星恒电源尽调项目中各 Agent 的实际协作链路——不是设计文档，是真实运行记录。

research ↔ Claude Code · 深度研究

research 完成 7 维度搜索 + KG 框架检索 → Claude Code 基于搜索结果做 Monte Carlo 10000 次模拟 + DCF 估值

暴露问题：v1 基于公开信息，营收高估约 100%（60-80 亿 → 实际 29.49 亿）

修正：v2 引入审计报告数据 + 天眼查交叉验证 → 营收修正至准确值

data ↔ Hermes · 研报四阶段管道

① 研报客API搜索+五维加权初筛 → ② 本地二级筛选（≥2MB/排除个股+周期点评/A/B/C三级分类）→ ③ OCR+LLM四维提取（MiniMax→sensenova→Qwen降级链）→ ④ 12项质量门禁+PG/Qdrant双写入库

暴露问题：初版无筛选标准→低质报告污染知识库；手工筛选仅覆盖13/108份高质量报告；API Key子进程检测误判

修正：四阶段管道体系化+Hermes V0→V3自进化（提取率30%→90%+）+系统扫描替代手工筛选（13→108份）

main → 4 子 Agent · 调度仲裁

用户指令 → main 拆解为 4 子任务 → research + data 并行启动 → project 跟踪 → report 收尾

暴露问题：research 两次 sessions_send 回报"已完成"，但文件实际未写入磁盘

修正：main 增加 fs 文件存在性校验 + 超时重试机制（最多 3 次）

project · 全流程纳管 + 线上部署

project 统一管理尽调底稿 → GitHub Pages 一键部署 → 桌面版 + 手机版双版本 → 链接直接发同事

暴露问题：初版无版本控制，尽调底稿分散在多个文件夹，同事找不到最新版

修正：project Agent 统一纳管 + GitHub 版本追溯 → jinzhanxiang.github.io/xingheng-report/

📐 协作时间线（真实运行记录）
05-14 16:46 首次启动 → 前 4 轮系统未校准 → 05-15 05:18 审计数据首次突破 → 05-15 当日 8 轮连续迭代 → 06-11→06-12 dd1→dd11 共 11 轮质量打磨（15 小时） → 06-20→06-22 Claude Code v1→v2→v3 纵深分析 → 07-02 最终交付

📌 关键发现：协作不是"设计出来的"，是踩坑踩出来的。每次 Agent 之间出问题（文件没写入、数据取错了、消息丢了），就加一道校验——30+ 次迭代后，协作链路才稳定。下一个项目启动时，这些坑不会再踩。

🦾 2 大外部助手 · 工程基建

🦾 Claude Code

职责：写代码 · 改代码 · 跑命令 · 做工程

核心能力：代码生成 · 重构 · 调试排错 · 命令执行 · 工程部署

与 main 的边界：
Claude Code 是执行工具，main 是调度者。
main 决定"做什么"，Claude Code 决定"怎么做代码"。

⚙️ Hermes

职责：脚本执行 · 自我迭代 · 复杂长链条任务攻关

核心能力：自我进化（V0→V3 四轮迭代，提取率 30%→90%+）· 多模型协作 · 飞书联动

定位：单兵最强的"特种兵"。超过 5 Agent 中任何一个的单独能力，擅长长线条深度任务。
天然短板：团队协作——擅长自己干，不擅长指挥别人干。
详见第 08 章 Hermes vs Claude Code 角色深化。

⚠️ 铁律：飞书群聊中 绝对禁止 @ 机器人，必须用 sessions_send 工具调用。否则机器人收不到通知。
