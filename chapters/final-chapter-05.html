第 02 章已用星恒项目的 4 个真实案例展示过幻觉。这里直接讲怎么防和重点关注什么。

📋 三道防线

1. 来源锁定

每条关键数据标注来源文件 + 段落。例："（FY24 审计报告 P23 营收 36.36 亿）"

2. 双重验证

关键数字必须两个独立源交叉验证——LLM 检索 + 外部 API，两源一致才采信。

3. self-check 自检

输出后三项检查：事实核查 / 逻辑一致性 / 来源可追溯。三项通过才算完成。

🔍 3 类高风险场景，重点关注

🔴 财务数字

营收、净利、估值倍数——最容易"迁移"。必须交叉验证两个独立源。

🟡 股权/股东信息

持股比例、关联方——LLM 倾向"合理推测"。必须工商登记核对。

🟣 Agent 状态声称

说"已完成"不代表真的完成。必须用文件校验而非信任消息。

📌 实测数据：引入三道防线后，幻觉率从 ~15% 降至 <3%。关键不是「更聪明的模型」，而是更严格的流程。

两个真实案例，同一个教训——加功能不等于变好，代码多不等于可靠。

🔴 案例 1：叠加质疑 Agent → 质量暴跌

晶能光电报告：叠加质疑 Agent 后 → 字数 -83%（6.5万→1.1万）、Insight -100%、数据点 -89%。

根因：新模块把原有分析步骤"吃掉"了。集成方式不对，新功能会破坏旧流程。

🟣 案例 2：64K 行代码 → 1 个方法文件

V3：64K 行 Python · 42 模块 · 9 步外挂式 QC 链 → 单向流水线，Phase 间断层。

改后：hv-analysis：0 行代码 · 2401 行方法文件 · 内嵌自检替代外部门禁。质量反超。

代码 ≠ 可靠
代码越多耦合越深，改一处牵动全身
质控要内嵌
让 Agent 写的时候就检查，不是写完了再打回
方法论 > 工程
给 Agent 一个好方法，胜过写一万行代码
📌 判断标准：如果你的 Agent 系统维护成本随时间上升——说明方法有问题。正确的范式：维护成本随时间下降，每次迭代都在沉淀原则而非堆积代码。完整对比见第 06 章 Demo 2。

📋 5 条踩坑教训 + 成本真相

⚠️ 先做对的事，再正确地做

坑 1：一开始就做模型微调 → 浪费 3 个月 → 先做编排，验证流程
坑 2：追求"完全自主 Agent" → 失控 → 人在回路，先做 80 分再迭代
坑 3：不做标准流程直接复制团队 → 报错 → 先沉淀 SOP 再培训
坑 4：忽略记忆功能 → Agent 失忆 → 每个 Agent 配独立记忆文件
坑 5：没有 main 调度 → 多 Agent 抢活 → 必须有一个总指挥

🔥 决策者一定会问的 5 个问题

从大量实际交流中提炼。每个问题有具体数据支撑，不回避不粉饰。

Q1 AI 真能写好深度报告吗？

单 Agent 不行，5 Agent 协作就能。传统 13 份研报 3 天 → 2 小时。主报告 ~64K 字，OCR 准确率 96.8%。

Q2 模型是不是越贵越好？

不是。三段式选型：80% 简单任务 → DeepSeek+GLM，15% 主力 → MiniMax M3，5% 复杂 → Claude Sonnet / DeepSeek V3/R1。

Q3 系统稳定性怎么保证？

3 层保障：Agent 互校验 → Wiki 检索交叉验证（5,894 实体 · 11,075 关系 · 193 份研报 · 99.7% uptime）→ 决策者最终判断。人是底线，AI 是放大器。

Q4 新人怎么快速入门？

5 阶段路径：手写 → AI 辅助 → AI 主导 → Agent 协作 → Agent 自治。新人入门从 3 个月 → 2 周。

Q5 AI 输出的东西我能直接用于决策吗？

不能，也不应该。AI 输出是参考资料，不是决策依据。正确用法：AI 提供数据、模型和初步分析 → 投资团队交叉验证关键数据 → 结合行业经验和商业判断做最终决策。AI 负责效率，人负责质量。所有数据必须可追溯到原始来源。
