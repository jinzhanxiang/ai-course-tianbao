🛡️ 我们踩过的坑 · 幻觉防御实录

LLM 会说谎——不是故意的，但结果一样严重。以下 4 个坑全部来自星恒电源项目的真实记录。先看坑，再看怎么防。

🔴 股东比例造假

LLM 声称某股东持股 17.73%，Tavily 交叉验证发现无此数据——数字从一个案例"迁移"到了另一个公司。

🟡 research 声称"已完成"

两次 sessions_send "已完成"，但文件实际未写入。main 通过 fs 校验发现差异并仲裁。

🟣 财务数字年份错配

星恒尽调中，LLM 将 2022 年营收 35.99 亿错误标注为 2024 年。原始数据源标记模糊导致。

🔵 SelfCheckGPT 一致性验证

10 个 LLM 输出做事实一致性检查，3/10 存在偏差。自检是第一道防线。

📋 cite-then-answer · 先引用再回答

1. 来源锁定

每条关键数据标注来源文件 + 段落。例："（星恒 FY24 审计报告 P23 营收 36.36 亿）"

2. 双重验证

关键数字必须两个独立源交叉验证。一个来自 LLM 检索，一个来自 Tavily / 外部 API。

3. self-check 自检

输出后三项检查：事实核查 / 逻辑一致性 / 来源可追溯。三项通过才算"完成"。

📌 实战效果：引入 cite-then-answer 后，幻觉类错误从 ~15% 降至 <3%。关键不是"更聪明的模型"，而是更严格的流程。

⚠️ 实战案例 · 叠加质疑Agent反而导致质量下降

以下是一个真实发生的"升级反而倒退"案例——在纵横分析法（HV Analysis）基础上叠加V3质疑Agent后，报告质量不升反降。

📐 基线：HV Analysis（纵横分析法）

实验对象：晶能光电（非上市LED芯片企业）
方法：HV Analysis V3.43.0 深度模式
架构：纵向（21年时间轴）+ 横向（6家可比公司）+ 交叉洞察
数据源：桌面文件 + iFind + AKShare + Tavily + arXiv API

🔬 增强版：+ V3 Challenge/质疑Agent

叠加模块：
① 数据溯源标注（源文件/来源类型/可信度A/B/C/D）
② 5级置信度校准（multi_source → contradiction）
③ Challenge B V3.0 防幻觉引擎（141数据点验证）
④ 论证/论点/论据三方质疑
预期：在保持基线质量的同时，每条数据有源可查

▼ 对比结果：预期提升 → 实际大幅下降

对比维度 基线（HV Analysis） 增强版（+ 质疑Agent） 差距
正文字数 65,265 字 11,136 字（MD） -83%
H2 章节 49 个 14 个 -35 个
Insight 洞察 11 个 0 个 -100%
数据点覆盖 445 个 ~50 个 -89%
PE 估值引用 13 处 0 处 -100%
横向对标公司 6 家 × 12 维度 1 节简略提及 -92%
来源标注率 部分标注 100%（141/141） ✅ 唯一提升项

🔴 根本原因：工作流被"腰斩"

指令："直接OCR阅读 + 不走data代理"
正确理解：不依赖外部数据库，只用本地文件
错误理解：不进行LLM深度分析，跳过工作流

被跳过的关键步骤：
• Step6 检索层（无横向对标数据采集）
• Step8 逻辑重组（无深度Insight生成）
• 纵向时间轴分析（无21年技术/财务时间线）
• 6维度对标模板（无结构化横向对比）

🟢 修复方案：本地文件模式≠跳过分析

代码级修复（hv_full_research.py）：
① 新增 --input-dir 参数指定本地文件目录
② 新增 local_context 注入到LLM prompt
③ 保留完整工作流：纵向→横向→交叉→估值→风险
④ 本地文件作为context（不替代分析步骤）

预期修复后：字数 30K+ | H2 30+ | Insight 8+

📌 这个案例的核心启示：AI Agent 的"升级"不是搭积木——不是把质疑Agent往上一叠就自动变好。新模块的加入会改变工作流的执行路径，如果集成方式不对，新功能会把原有功能"吃掉"。真正有效的做法是：在框架层面（代码参数 + prompt注入）集成新模块，而不是在流程外做后置处理。这和你的日常工作没有本质区别——引入新工具要先改造流程，而不是把新工具硬塞进旧流程里。

🧬 案例 · 研究方法的范式转移：从「管Agent」到「帮Agent」

以下案例发生在2026年7月6日——从发现问题到完成范式转换，仅用了一天。这不是计划好的升级，而是一次「架构诊断→认知转变→方法重构」的完整闭环。

🔴 阶段一：V3 的结构性缺陷

17 phase × 70 模块 × 64K 行 Python
• Phase 1 输出被 Phase 2 忽略（架构断层）
• 数据源路由刚性预设（MX → Qdrant → KG 必须全走）
• 单向 12 步，不能回退补充检索
• 质疑节点是外部门禁（Agent 不知道为什么被打回）
核心问题：64K 行代码实现的是一个僵硬的流水线

🟡 阶段二：关键认知转折

一个朴素的问题：
为什么老分析师比年轻分析师强？

不是因为「步骤更多」或「模板更全」：
① 知道什么时候该用什么数据源
② 能识别数据缺口并主动补充
③ 会交叉验证而不是照单全收
④ 经验是原则而不是清单
核心认知：数据源应该服务 Agent，而非框死 Agent

🟢 阶段三：纵横分析法诞生

1 个 SKILL.md，0 行代码
• Phase 0 前置：先读本地一手文件，再搜
• 自适应闭环：每章 Q1-Q4 自检 → 补缺口 → 循环
• 数据源路由规则：告诉「用什么/禁止用什么」，不规定调用顺序
• 警醒信号 S1-S7：教 Agent「如何发现自己的偏见」
• 降级链：MX → iFind → SearXNG（失败不退出）
核心公式：原则 > 清单，自检 > 门禁
维度 V3 工作流 纵横分析法
定位 给流水线的配置文件（17 phase × 70 模块） 给 Agent 的方法论（1 个 SKILL.md）
数据源 刚性路由：固定调用 MX → Qdrant → KG 自适应路由：Agent 识别缺口后自主选择
反馈 单向 12 步，不可回退 闭环循环：每章自检 → 补缺口
质控 质疑节点 A/B（固定阈值 60%/70%，外部门禁） Q1-Q4 自检 + S1-S7 警醒信号（内嵌自检）
代码量 64K 行 Python 0 行（纯方法论文件）
输出 自由格式 25 章固定模板（研究+决策）

✅ 晶能光电实战验证

方法报告 vs 基线报告（人工+AI 协作）：
• 总字数：27,629 vs 14,586（+89%）
• 数据来源标注：49 处、28 种 vs 少数
• 自检覆盖：26 处 Q1-Q4 vs 无
• 警醒信号：S1-S7 全激活 vs 无
• 方法报告独有 6 个决策维度（基线缺失）：
交易结构分析 · 退出路径分析 · 约束条件建模 · 情景推演+IRR分级 · 6维竞品对标 · 估值三锚校验

💡 4 个关键启示

1. Agent 需要「原则」而不是「清单」
清单越细，Agent 越蠢。给它路由规则+自检问题+降级链，让它自己判断。
2. 数据源为 Agent 服务，不是反过来
Agent 读本地文件→识别缺口→选择工具→循环。数据源是可选工具，不是必经之路。
3. 质控是「自检」不是「门禁」
不是「你写完了我来审」，而是「你写的时候问自己四个问题」。
4. 方法论演进速度快于工程实现
3 段文字、2 个模板 = 数周的工程排期。方法论就是产品。

📌 这个案例的核心启示：在 AI 辅助投资研究领域，最高杠杆的改进点不是工程自动化，而是给 Agent 一个好方法。从 64K 行 Python 到 1 个 SKILL.md，代码量降为 0，输出质量系统性超越基线——这就是方法论的力量。这和你的日常工作一样：工具很重要，但方法论才是天花板。

🫀 HEARTBEAT 心跳机制

Main 代理每 15 分钟执行一次全局心跳检查，覆盖 9 项清单：

1. Docker 服务状态（PostgreSQL + Qdrant）
2. 任务跟踪（in-progress > 2h 告警）
3. 异步任务超时重试（最多 3 次）
4. Session 状态巡检（> 3MB 自动压缩）
5. 内存使用监控
6. 配置文件审定
7. 知识库一致性校验
8. API 配额余量检查
9. 日志异常模式扫描

🛡️ 多系统协同维护

多个智能体系统并行运行时的关键维护点：

端口管理主控平台: 18789 · Hermes: 独立端口
API 网关统一 token 管理 · 多模型兼容
免打扰窗口23:00-08:00 静默但继续检查
告警升级pending → in-progress → overdue 三级

⚠️ 维护原则：免打扰 ≠ 不检查。23:00-08:00 期间心跳照常执行，异常写入 pending-alerts.json，解静默后统一汇报。Session 超过 3MB 自动触发压缩，兼顾性能与上下文保留。

🛠️ 今日实战 · Claude Code 系统维护验证（2026.07.05）： OpenClaw 尝试自升级未完成（2026.6.8 → 2026.6.11 待升级）→ Claude Code 检测 → 诊断（无失败日志，是根本没执行升级动作）→ 执行升级（19.34s，含 doctor 校验）→ 验证版本号 → 重启网关 → 2 插件更新 + 3 缺失插件自动补装（kimi/searxng/tavily）。AI 维护 AI 基础设施，不用人操心。

💡 这对你的日常工作意味着什么：你早上 9 点打开飞书，main 已经在昨晚自动完成了 9 项系统检查。你看到的消息是「系统正常，3 个 Agent 在线」——而不是「某某服务挂了，请手动重启」。你不用管运维，你只管用。
