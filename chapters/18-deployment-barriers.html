🛠️ 🛠️ 门槛 1 · 安装部署 · 主公破解法

痛点：在线产品很多（Claude/GPT/DeepSeek），但本地部署才是关键 — 数据不外泄 + 隐私可控 + 离线可用

🔑 破解思路 A：工具代理安装

Step 1：先装一个工具类智能体
→ 推荐：Claude Code（命令行式，最易上手）
→ 备选：DeepSeek 在线（国内访问稳定）

Step 2：用这个工具协助安装其他智能体
→ 一行命令：claude "帮我装 ollama + omlx"

效果：原本 2 天的安装 → 30 分钟完成

🔁 破解思路 B：智能体互装

核心思路：装完一个 → 让它装第二个 → 第二个装第三个...

真实路径（主公）：
① Claude Code 装 ollama（云端大模型）
② ollama 拉 Qwen3.6-27B 本地模型
③ 用 ollama 装 omlx（Apple Silicon 专属）
④ omlx 协助装 OpenClaw 整套智能体体系

效果：1 人 1 周 → 1 人 1 天 完成 5+2+1+1 体系

📋 主公实战 4 步安装链

# Step 1: 安装 Claude Code（工具）
npm install -g @anthropic-ai/claude-code
claude "请帮我安装 ollama"

# Step 2: 装 ollama（云端大模型管理）
curl -fsSL https://ollama.com/install.sh | sh
ollama pull qwen3.6:27b

# Step 3: 装 omlx（Apple Silicon 端侧推理）
# （详见 https://github.com/jinzhanxiang/omlx）
ollama run qwen3.6:27b "帮我装 OpenClaw"

# Step 4: 装 OpenClaw 5+2+1 体系
openclaw init --arch "5+2+1+1"
openclaw gateway start

🔧 🔧 门槛 3 · 应用调试 · 智能体协作维护

痛点：智能体不是装完就完事 — 需要更换模型、修复问题、升级版本，传统 IT 维护模式不再适用

🤖 路径 A：Claude Code 维护

典型场景：MiniMax API 报 429 限流

主公命令：
claude "MiniMax 限流了，帮我把 fallback 切到 DeepSeek"

Claude Code 动作：
① 读 openclaw.json 配置
② 改 fallbacks 数组
③ 热重载 Gateway
④ 验证 fallback 生效

耗时：15 分钟（人工需 1 小时）

🤝 路径 B：智能体互维护

典型场景：data 代理 OCR 准确率下降

多智能体协作：
① data 发现 OCR 错误率升 → 通知 main
② main 自动调用 research 分析
③ research 诊断：模型版本过旧
④ data 自动升级 Qwen-VL-Max 版本
⑤ report 生成维护报告

耗时：零人工干预 · 全自动

🌱 🌱 门槛 4 · 日常培育 · 让智能体越用越聪明

核心认知：智能体是"会进化的工具"，不是一劳永逸的产品。3 个月不培育 = 落后 1 代模型；6 个月不维护 = 完全过时

① 应用场景驱动

误区："为用智能体而用"
正路：从业务痛点出发

主公 4 个真实场景：
① 投研尽调（星恒 13.73%）
② 文档修复（晶能 V3426）
③ 飞书协同（5+2+1 调度）
④ 数据清洗（200+ 研报）

每个场景都有基线

② 设置基线 + 量化

主公 5 个核心基线：

① 准确率：OCR ≥96%
② 速度：尽调 ≤2 小时
③ 成本：≤1.5 分/次
④ 稳定：心跳 99%+
⑤ 错误率：≤2%

不达标 = 自动回滚

③ 逐步迭代

主公 4 步迭代法：

① 小流量灰度（A/B 5%）
② 对比基线（每 100 次）
③ 全量上线（达标）
④ 记录知识（MEMORY.md）

🌳 4 大门槛的相互关系

① 安装部署是起点（地基）→ 没有装好 = 一切免谈
② 模型费用是约束（天花板）→ 选错模型 = 业务不可持续
③ 应用调试是核心（发动机）→ 不调试 = 模型再好也用不起来
④ 日常培育是关键（续航）→ 不培育 = 3 个月后落后一代

主公核心心法：装得起的用得起 · 用得起的调得动 · 调得动的养得活
