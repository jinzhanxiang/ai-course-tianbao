🎯 一句话选型铁律

云端 = 高质量日常 · 本地 = 隐私 + 兜底 · oMLX = Apple Silicon 专属 + MLX 框架

☁️ 云端路线

适合：日常高质量对话、代码生成、复杂推理
主公配置：MiniMax-M3（primary）→ M2.7（fallback1）→ Kimi-for-coding（fallback2）
特征：质量最高 / 速度稳定 / 需联网 / 隐私风险

🖥️ 本地路线

适合：隐私数据处理、云端故障兜底、批量嵌入
主公配置：oMLX (port 8080) + Qwen3.6-27B / bge-m3
特征：零数据外泄 / 离线可用 / 受硬件限制

🛡️ 兜底逻辑

适合：云端 429、本地 503、网络中断
主公配置：LiteLLM fallback 自动切换
特征：主公 39.3 万请求跑出来的实战经验

🍎 Apple Silicon / MLX · 为什么主公选这条路

MLX 是 Apple 2023 年发布的专为 Apple Silicon 设计的 ML 框架 —— 与 llama.cpp/ollama 有本质差异

⚡ Zero-copy 架构

CPU 和 GPU 共享统一内存（unified memory），无需 cudaMemcpy 式数据搬运 —— 这是 Apple 设备上的核心优势

🧠 M5 Neural Accelerator

2025 年起 MLX 自动利用 M5 芯片的矩阵乘法加速器 —— 后续设备性能会进一步提升

💾 M3 Ultra 192GB 显存

可直接装 70B 模型无需 quantization，70B Q4 在 M3 Ultra 上实测 25-30 tok/s

📈 学术印证

vllm-mlx（arXiv 2026）在 Qwen3 / Nemotron 上 比 llama.cpp 吞吐高 21-87%，M4 Max 峰值 525 tok/s

🎯 实战结论：如果你是 Apple Silicon 用户（M1+），omlx 是 ollama 的超集—— ollama v0.9+ 也在用 MLX backend，间接说明 MLX 是 Apple 端的事实标准

⚠️ 实战中暴露的真实问题

❌ 已知痛点

  • 大模型幻觉（5+2+1 层叠可减不灭）
  • oMLX 并发限 3（max_concurrent_requests）
  • 本地 Qwen3.6-27B vs 云端 M3 质量差距
  • Wiki 数据清洗 328 个脚本累积复杂
  • Hermes + Claude Code 协同偶发掉线

✅ 兜底方案

  • 云端 429 → 自动 fallback 到本地
  • 本地 503 → LiteLLM 切回云端
  • 云端 M3 限流 → 切 M2.7（次优）
  • M2.7 限流 → 切 Kimi-for-coding
  • 全部失败 → 进入 waiting 队列，cron 重试
💡 兜底设计的核心哲学：永远不要让"我本地用不了"成为阻塞 —— 至少有一条路能跑通。主公的 LiteLLM fallback 让 39.3 万请求零丢失

🌳 部署选型决策树

Q1. 数据涉密？

✅ 是 → 本地路线（omlx / ollama）

❌ 否 → 继续 Q2

Q2. 设备是 Apple Silicon？

✅ 是 → omlx 优先（零拷贝，吞吐最优）

❌ 否 → 继续 Q3

Q3. 单请求质量要求高？

✅ 是 → 云端 + ollama 兜底（云端为主，ollama 兜底）

❌ 否 · 高并发 → vllm 集群（生产级 serving）

🎯 主公的答案：Apple Silicon + 数据敏感 → omlx 主 + 云端 fallback（39.3 万请求验证）
