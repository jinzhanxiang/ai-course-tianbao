🔄 四个阶段 · 从启动到交付的完整链路

智能体不是「一键出报告就完事了」——以下是星恒电源项目从第一版到最终交付的真实路径。

🟡 Phase 1 · 05.14 首次启动（前 4 轮系统校准）

💡 人的判断：拿到 v1 报告后，立即发现营收数字与其他尽调材料矛盾 → 触发审计数据修正流程。

🟡 Phase 2 · 05.15 审计突破 + 当日 8 轮迭代

💰 估值计算全链路 · 从数据到结论

这不是「AI 拍脑袋给了一个估值」——以下展示完整的计算链路，每一步都可追溯。

📐 计算链路

1. 历史基准 IRR 模型（复合增长率 0-15% / 净利率 1.5-2.5% / PE 8-12倍）
2. → 全部组合深度亏损
3. → Monte Carlo 10000 次模拟
4. → P10/P50/P90 全分布
5. → 估值 36.4 亿，较收购价亏损约 68%

🔑 关键迭代

• v1 基于公开信息 → 营收高估约 100%（60-80 亿 → 实际 29.49 亿）
• v2 引入审计数据 → 营收修正至准确值
• v3 产业链上下游全景分析 → 竞争格局量化

📊 数据交叉验证清单

✅ 营收：审计报告 29.49 亿 vs 天眼查 29.5 亿（吻合）
✅ 净利：审计报告 2,163 万 vs 行业均值 ~2,000 万（吻合）
✅ 股东：天眼查 50 家 vs 工商登记 50 家（100% 覆盖）
✅ 竞品：宁德时代/比亚迪/中创新航 12 维度对标
⚠️ 产能规划：管理层声称 vs 实际产出偏差 67%
AI 辅助可大幅压缩计算时间。但估值假设的合理性仍需投资团队确认——AI 算数，人做判断。

🛡️ 踩过的坑 · 幻觉防御实录

LLM 会产生事实错误（幻觉）——不是故意的，但结果一样需要警惕。以下 4 个坑全部来自星恒电源项目的真实记录。

🔴 股东比例造假

LLM 声称某股东持股 17.73%，Tavily 交叉验证发现无此数据——数字从一个案例"迁移"到了另一个公司。

🟡 research 声称"已完成"

两次 sessions_send "已完成"，但文件实际未写入。main 通过 fs 校验发现差异并仲裁。

🟣 财务数字年份错配

LLM 将 2022 年营收 35.99 亿错误标注为 2024 年。原始数据源标记模糊导致。

🔵 SelfCheckGPT 一致性验证

10 个 LLM 输出做事实一致性检查，3/10 存在偏差。自检是第一道防线。

📋 cite-then-answer · 先引用再回答

1. 来源锁定

每条关键数据标注来源文件 + 段落。例："（星恒 FY24 审计报告 P23 营收 36.36 亿）"

2. 双重验证

关键数字必须两个独立源交叉验证。一个来自 LLM 检索，一个来自 Tavily / 外部 API。

3. self-check 自检

输出后三项检查：事实核查 / 逻辑一致性 / 来源可追溯。三项通过才算"完成"。

📌 实战效果：引入 cite-then-answer 后，幻觉类错误从 ~15% 降至 <3%。关键不是「更聪明的模型」，而是更严格的流程。

🧬 方法的进化：从「管 Agent」到「帮 Agent」

星恒项目不仅交付了一份报告，更催生了一个方法论突破——从 64K 行 Python 的刚性流水线，到 1 个方法文件的自适应方法论。

🔴 旧范式：V3 刚性流水线

• 17 phase × 70 模块 × 64K 行 Python
• 数据源路由刚性预设
• 单向 12 步，不能回退补充检索
• 质疑节点是外部门禁
核心问题：64K 行代码实现的是一个僵硬的流水线

🟢 新范式：纵横分析法（hv-analysis）

• 1 个方法文件，0 行代码
• 自适应闭环：每章 Q1-Q4 自检 → 补缺口 → 循环
• 数据源路由规则：告诉「用什么」，不规定调用顺序
• 警醒信号 S1-S7：教 Agent 「如何发现自己的偏见」
核心公式：原则 > 清单，自检 > 门禁
📌 这个案例的核心启示：在 AI 辅助投资研究领域，最高杠杆的改进点不是工程自动化，而是给 Agent 一个好方法。从海量代码的刚性流水线，到一份方法文件驱动的高质量输出——这就是方法论的力量。这和日常工作一样：工具很重要，但方法论才是天花板。
