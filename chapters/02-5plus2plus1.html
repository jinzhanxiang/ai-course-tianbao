🤖 什么是智能体？

先看清楚这场变革 —— 智能体时代 vs 传统时代，到底变了什么？

变革大背景：我们正在经历 「软件 2.0」 的变革。从「手工生产」→「智能工厂」需要三步：云计算 (2010s) → 大模型 (2023) → 智能体 (2024+)。

📊 智能体时代 vs 传统时代（5 个核心差异）

维度 传统软件时代（2000-2023） 智能体 Agent 时代（2024+） 变革
交互方式 点按钮 / 填表单 / 下拉菜单 说话下达目标 从 UI 到 NL
使用主体 人被软件指挥 人指挥软件干活 主体反转
输出类型 信息展示 / 表格汇总 执行结果 / 交付物 从查到做
调用关系 单个软件独立运行 多个 Agent 并行协同 + 反向调用人 从点到网

🎯 5+2+1+1 解决了什么？

❌ 痛点 1：一个人干不了

投研需要 6 个专业能力：行业 + 财务 + 估值 + 项目管理 + 数据清洗 + 报告排版。一个人 6 年也学不全。

→ 5 Agent 分工：main 调度、research 分析、data 清洗、project 管理、report 排版

❌ 痛点 2：工具链太复杂

🍎 Apple Silicon / MLX · 为什么主公选这条路

MLX 是 Apple 2023 年发布的专为 Apple Silicon 设计的 ML 框架 —— 与 llama.cpp/ollama 有本质差异

⚡ Zero-copy 架构

CPU 和 GPU 共享统一内存（unified memory），无需 cudaMemcpy 式数据搬运 —— 这是 Apple 设备上的核心优势

🧠 M5 Neural Accelerator

2025 年起 MLX 自动利用 M5 芯片的矩阵乘法加速器 —— 后续设备性能会进一步提升

💾 M3 Ultra 192GB 显存

可直接装 70B 模型无需 quantization，70B Q4 在 M3 Ultra 上实测 25-30 tok/s

📈 学术印证

vllm-mlx（arXiv 2026）在 Qwen3 / Nemotron 上 比 llama.cpp 吞吐高 21-87%，M4 Max 峰值 525 tok/s

🎯 实战结论：如果你是 Apple Silicon 用户（M1+），omlx 是 ollama 的超集—— ollama v0.9+ 也在用 MLX backend，间接说明 MLX 是 Apple 端的事实标准

Phase G 新增章节 · 2026-07-03 · 主公实测

金句开篇

"模型一定会撒谎，但人可以不接。 cite-then-answer 让你看到证据再说话， self-check 让你对自己的回答二次审问。"

—— 主公 2026-07-03 实战方法论

📌 📌 方法 1：cite-then-answer（先引用，再回答）

核心范式：让 LLM 在生成回答前，必须先引用给定的参考资料（RAG 上下文）。把"模型靠记忆自由生成"转为"模型被迫从给定证据中取词"。

🔧 主流实现

  • Anthropic Citations API（2025-01）
    用户上传 PDF → 自动 chunk → 绑定 source
  • OpenAI File Citations
    file_citation 工具调用返回引用
  • LangChain / LlamaIndex
    with_structured_output 返回 answer+citations

✅ 实战 prompt 模板

[System]
你是一个严格的研究助理。回答时必须按以下流程：
1. 阅读我提供的 reference 文档
2. 每句结论必须以 [ref:N] 标注来源句
3. 如果 reference 中找不到对应答案，明确回答"未在 reference 中找到"
4. 严禁补全、推断或编造 reference 之外的事实
💡 主公应用：research V3 的 Reference 强制引用机制 —— 任何输出必须带 [ref:N] 标签，否则视为不通过

🔍 🔍 方法 2：self-check（自检 prompt）

核心范式：让模型在完成主任务后，对自己的输出做"二次审阅"。本质是 LLM-as-a-Judge，但目标不是评估而是自我纠正。

① 事实核查

审查以下回答，逐一标出每个事实句为：
- ✅ 正确（有依据）
- ⚠️ 需要验证（依据不足）
- ❌ 错误（与事实不符）

② 逻辑一致性

检查以下回答的内部逻辑：
1. 前提 A 是否被支撑？
2. 推论 B 是否由前提得出？
3. 结论 C 是否与推论一致？
列出 1-3 个具体问题。

③ 来源追溯

对每条结论，要求回答：
1. 结论内容（必须基于 source）
2. 引用句位置（哪个 source、哪个段落）
3. 信心等级（高/中/低）
⚠️ 已知局限：self-agreement bias —— 模型会"自我欣赏"。同一权重检查同一输出，会发现自己很棒。**所以需要"多 Agent 辩论"（见 11 章）+ 外部 source 强制 cite**

⚔️ ⚔️ 方法 3：多 Agent 辩论（呼应第 11 章）

核心思想：单 Agent 自检的 60-70% 准确率无法突破 —— 必须用多 Agent 互检。详见第 11 章「智能体质量提升」，本章给主公实战架构。

🟢 正方 Agent

角色：作为"原始结论的拥护者"
任务：解释为什么这个结论是对的
模型：M3（最强）

🔴 反方 Agent

角色：作为"挑刺者"
任务：列出这个结论的 3 个潜在错误
模型：M2.7（独立判断）

🟣 仲裁 Agent

角色：作为"裁判"
任务：综合正反方，给出最终结论
模型：M3（重新读所有）

🎯 实战数据：主公实测 — 单 Agent 自检 60-70%，多 Agent 辩论 85-90%。差距20+ 百分点，是结构性优势

🎯 主公实战方法：4 步流程

Step 1 · cite-then-answer —— 先给 reference，强制 [ref:N] 标注

→ 失败的输出：99% 都缺 reference

Step 2 · 事实核查 self-check —— 对主输出逐句标 ✅/⚠️/❌

→ ❌ 的句子必须改写或删除

Step 3 · 多 Agent 辩论 —— 正方/反方/仲裁 三角对抗

→ 反方必须能列出 3 个潜在错误，否则视为正方通过

Step 4 · fs 校验 —— 落地文件必须 ls/wc/grep 三重验证

→ "口头完成"无效，必须磁盘文件存在且完整

📌 关键纪律：研究类任务必须 先 write 再 announce。announce 步骤 = "磁盘文件 + 业务验证"双重确认

🚫 5 个常见避坑

❌ 1. "模型说应该这样" = 接受

主公不接"看起来对" —— 必须有 reference 标注

⚠️ 2. self-check 后 self-agree

模型会"自我欣赏"，必须多 Agent 互检，不能自检自

⚠️ 3. 量化模型更易幻觉

Q4/Q8 量化后数字精度下降，财务数据禁用本地小模型

💡 4. RAG ≠ 万能

如果 reference 本身就错，RAG 也会"自信引用"错误信息

✅ 5. "口头完成" = 没完成

fs 校验失败 = 任务失败，必须重做

💡 6. 频率 vs 深度平衡

5 次自检 = 5 倍时间；多数场景 1 次 + 多 Agent 辩论够用
