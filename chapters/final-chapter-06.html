🔴 Demo 2A · 先看看这套"看起来很严谨"的架构

4 Phase 流水线 × 9 步质控链 · 64K 行 Python · 42 个模块 · 这是真实在跑的 V3 系统

📐 4 Phase 流水线（单向，无反馈回路）

Phase 0 · 识别与分派 判断任务类型 → 识别公司类型 → 选择数据源
Phase 1 · 并行数据采集 4 个 Work Stream 同时启动：财务 / 商业 / 法务 / 运营
Phase 2 · 并行单流分析 每个 WS 独立推理：财务评分 / 竞争格局 / 合规风险 / 运营风险
Phase 3+4 · 交叉合成 + 报告生成 串行等待上游 → 估值/风险/协同 3 项交叉 → 按框架输出报告

🔍 每个 Work Stream 内部：9 步质控链

数据层：质量过滤 → 逻辑推理 → 覆盖率检查
验证层：一致性验证 → Challenge A 评分 → 子框架覆盖度
输出层：关键发现提取 → 非上市对标 → 综合置信度

⚙️ 运行数据

64K+ 行 Python · 42 个模块 · 全流程约 16 分钟
Challenge A 通过率 >90% · 报告完整度 >95%
🔍 点击展开 · 看每个 Phase 的完整步骤

Phase 0 · 识别与分派（4 步）：
Step 0.1: 任务类型识别 → 选择主框架（并购尽调/股权投资/行业研究）
Step 0.2: 公司类型识别 → listed/unlisted → 决定数据源策略
Step 0.3: 主框架分解 → 3-5 个 Work Stream，每个附带子框架模板
Step 0.4: 数据源策略 → 根据公司类型预设 MX/Qdrant/KG 调用顺序

Phase 1 · 并行数据采集（5 步/WS）：
Step 1.1: 数据源调用 → MX 财务指标 / 天眼查工商 / SearXNG 新闻 / KG 知识检索
Step 1.2: 数据完整性检查 → 必填字段覆盖率 ≥ 90%
Step 1.3: 数据质量过滤 → 时效性/来源权威性/重复检测
Step 1.4: 缺失数据标记 → 标记为"待补充"并记录原因
Step 1.5: 原始数据归档 → 存入 PostgreSQL 原始表

Phase 2 · 并行单流分析（4 步/WS）：
Step 2.1: 单维度推理 → 财务评分/竞争格局/合规风险/运营风险
Step 2.2: Challenge A 自评 → 60% 阈值，低于则触发重采
Step 2.3: 非上市对标 → 行业均值/可比公司/历史区间
Step 2.4: 关键发现提取 → 每个 WS 输出 Top 3 发现

Phase 3+4 · 交叉合成 + 报告生成（3 步）：
Step 3.1: 串行等待全部 Phase 2 完成 → 无反馈回路
Step 3.2: 3 项交叉验证（估值/风险/协同）→ Challenge B 综合 ≥ 70%
Step 3.3: 按框架模板生成报告 → 自由格式输出

关键缺陷：Phase 1→2 单向不可回退（Phase 1 数据不足时 Phase 2 不知道），质控 A/B 是外部门禁（Agent 不知道为什么被打回），数据源路由刚性预设（不管缺什么数据，都走同一套调用顺序）。

看起来很严谨，对吧？
但问题就出在"严谨"上——Phase 间断层、质控是外挂的、Agent 不知道"为什么被打回"。

🟢 Demo 2B · 但是——严谨 ≠ 好用

上一页展示了 V3 的全部架构细节。下面看问题出在哪，以及怎么用1 个方法文件反超——并在星恒电源项目上验证。

V3 架构
hv-analysis

🔴 阶段 1：V3 工作流搭建

· research Agent 搭建 4 Phase 流水线
· 53 个 Python 文件 · ~64K 行代码
· 19 固定维度 · Challenge Gate A/B 质控
· 40+ 版本迭代（V3.19 → V3.41.11）
问题：5 类 bug 反复修复 20+ 次

🟡 智能编程助手 诊断 3 大缺陷

① Phase 间断层：Phase 1 推理结果被 Phase 2 忽略，无法回退补充检索
② 外挂式质控：9 步 QC 外部执行，Agent 不知道「为什么被打回」
③ 交叉合成太浅：仅 3 项交叉，WS1 结论不触发 WS2 重新审视

🟢 阶段 2：hv-analysis 升级

· 智能编程助手 主导：识别 V3 模式缺陷 → 提出方法论替代方案
· 1 个 SKILL.md（2401 行）替代 64K 行代码
· 4 Phase 自适应闭环（每章 Q1-Q4 自检 → 即时补检索）
· S1-S7 警醒信号（决策偏见检测）
· cite-then-answer 幻觉率 15% → <3%
一天内完成认知跃迁

🔵 阶段 3：星恒项目方法论验证

· 项目：天保-星恒电源尽调分析
· 16 份报告 · 主报告 ~64,000 字
· 多源数据交叉验证（PS/PB/PE 三方法）
· 四层护城河侵蚀追踪（FY2020-FY2025）
· 6 家公司 31 个法律问题核查
· 第 02 章已详细展开项目全貌

📊 方法论改进效果

旧方法（V3）：报告质量得分 75.5/100，幻觉率 ~15%
新方法（hv-analysis）：内嵌自检替代外部门禁
cite-then-answer 防线：幻觉率降至 <3%
结论：方法文件的改进直接提升报告质量
对比维度 V3 旧范式 hv-analysis 新范式
代码量 64K+ 行 Python · 42+ 模块 0 行代码 · 1 个 SKILL.md（2401 行）
架构 Phase 1/2 并行，Phase 3 串行等待全部上游，无反馈回路 4 Phase 自适应闭环，每章 Q1-Q4 自检 → 即时补检索
质控 9 步 QC 链外挂式门禁，Challenge A 4 维 ≥40 / B 7 维 ≥75 内嵌自检 Q1-Q4（写时自查）+ S1-S7 警醒信号（决策偏见检测）
深度 4 WS 独立分析，但交叉合成仅 3 项，缺产业链纵深 21-25 章自适应模板，纵+横交叉洞察 7 章，含产业链全景
幻觉率 ~15%（年份错配/数字迁移） <3%（cite-then-answer + 双重验证）
估值 单一 DCF，无模拟 Monte Carlo 10000 次 + IRR 双模型交叉
输出质量 ~8,000 字，需人工大量补充 21,392 字，信息密度 2.6x，可独立交付
📌 核心公式：原则 > 清单，自检 > 门禁。64K 行代码实现的工程自动化，被 1 个方法文件反超——最高杠杆的改进点不是写更多代码，而是给 Agent 一个好方法。research Agent 做广度（搭建工作流+执行研究），智能编程助手 做深度（识别架构缺陷+设计方法论+验证估值模型）。
