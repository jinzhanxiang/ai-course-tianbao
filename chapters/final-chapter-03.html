🏗️ 先看一个智能体的最小架构

一个智能体 = 大脑 + 双手 + 记忆 + 计划 —— 缺一不可

🧠

大脑（大模型）

负责推理、决策、规划。MiniMax / Kimi / Claude / DeepSeek / 小米 Mimo，多模型融合。

🔧

双手（Tools）

读 PDF、写文件、跑 Python、调数据库。Function Call 协议。

📝

记忆（Memory）

短期上下文 + 长期向量库。本体系用 Qdrant + bge-m3。

📋

计划（Planning）

CoT 思维链 + ToT 思维树 + ReAct 推理-行动循环。

这套最小架构落到投资行业，自然扩展为 5+2+1+1 体系：
投资工作流天然分工——调度→研究→数据→项目→报告，每个环节由独立 Agent 负责。外加两个工程助手、一个知识中台和一个本地模型。

🏗️ 为什么是 5 个独立智能体，而不是 1 个派生 4 个？

一个常见的替代方案：单个智能体临时派生四个子代理。为什么没有选这条路？

❌ 方案A：单智能体 + 临时派生

• 子代理临时派生，任务结束即销毁
• 无法设置身份：没有身份文件 / 角色文件
• 无记忆积累：每次从零开始，不会越用越强
• 上下文压力集中：主 Agent 承载所有子任务上下文
⚠️ 核心缺陷：无身份、无记忆、无成长。就像每次任务都招一批临时工。

✅ 方案B：5 个智能体 + 会话调用

• 独立身份：每个智能体有身份文件 · 角色文件 · 工具文件 · 记忆文件
• 专业技能沉淀：各自强化，不会互相污染
• 记忆持续积累：每次任务自动入库，检索命中率逐月提升
• 上下文隔离 × 并行执行：5 个智能体 = 5 倍有效上下文
✅ 核心优势：持久身份 + 独立记忆 + 技能积累。每个人越做越强。

⚙️ 系统维护 · 心跳监控与多系统协同

5 Agent + Hermes + Docker 服务 + 知识库 —— 多系统长期运行需要体系化的监控与维护

🫀 HEARTBEAT 心跳机制

Main 代理每 15 分钟执行一次全局心跳检查，覆盖 9 项清单：
1. Docker 服务状态（PostgreSQL + Qdrant）
2. 任务跟踪（in-progress > 2h 告警）
3. 异步任务超时重试（最多 3 次）
4. Session 状态巡检（> 3MB 自动压缩）
5. 内存使用监控 · 6. 配置文件审定
7. 知识库一致性校验 · 8. API 配额检查
9. 日志异常模式扫描

🛡️ 关键维护机制

免打扰窗口23:00-08:00 静默但继续检查
告警升级pending → in-progress → overdue 三级
异常处理写入 pending-alerts.json，解静默后统一汇报
Session 管理> 3MB 自动触发压缩，兼顾性能与上下文
💡 这对你的日常工作意味着什么：你早上 9 点打开飞书，main 已经在昨晚自动完成了 9 项系统检查。你看到的消息是「系统正常，3 个 Agent 在线」——而不是「某某服务挂了，请手动重启」。你不用管运维，你只管用。
